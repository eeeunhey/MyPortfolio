  • 응답 수신 시 fade-in 전환으로 자연스러운 전환 구현

PROJECT 02

BalanceEat

2025.10–2025.11 · 4인 팀 · PO / Frontend
ReactTypeScriptGemini AINutrition APIFigma
먹은 음식의 영양소를 자동으로 분석하고, 부족한 영양소에 맞춰 AI가 식단을 추천하는 서비스를 구현했습니다.

Problem

  • 기존 식단 앱은 음식을 일일이 수동 입력해야 해서 사용자 부담이 컸습니다
  • 영양을 분석해도 맞춤 추천까지 이어지지 않았습니다
  • 어떤 영양소가 부족한지 사용자가 직접 판단하기 어려웠습니다

Solution

  • 음식 이름을 입력하면 Nutrition API로 영양 정보를 자동 검색하도록 연동했습니다
  • 3일치 섭취 데이터를 Gemini AI에 전달해 부족 영양소 기반 식단을 추천하도록 구성했습니다
  • 영양소별 달성률을 도넛 차트로 실시간 표시하도록 설계했습니다

Key Features

  • AI 식단 추천 — 3일간 섭취를 분석해 부족 영양소 기반 식단을 제안하도록 설계했습니다
  • 달성률 차트 — 칼로리·단백질·탄수화물·지방별 목표 달성률을 도넛 차트로 구현했습니다
  • 날짜별 기록 — 캘린더에서 날짜를 선택하면 식단과 영양 요약을 조회하도록 구성했습니다
  • 타입 안전성 — TypeScript interface로 API 응답 구조를 사전 정의해 오류를 방지했습니다
  • 오프라인 저장 — Local Storage를 활용해 서버 없이 데이터가 유지되도록 구현했습니다

개발 중 겪은 문제

문제
  • Nutrition API 응답 구조가 음식마다 필드가 다름 (칼로리만 있고 단백질 없는 경우)
  • 누락된 필드를 차트에 넘기면 undefined 에러 발생
  • 영양소 4개 도넛 차트가 데이터 변경 시 전체 리렌더링
해결
  • 기본값(0) 정규화 함수로 누락 필드 대응
  • TypeScript interface로 영양소 타입 사전 정의 → 컴파일 시점에 오류 감지
  • React.memo + useMemo로 변경된 영양소만 다시 그리도록 최적화

사용자 경험 흐름

음식 검색→ 영양 자동 표시→ 달성률 차트→ AI 식단 추천→ 날짜별 기록 조회

음식 이름만 입력하면 영양 정보가 자동으로 표시되고, 하루 목표 대비 달성률을 차트로 확인할 수 있습니다. 3일 이상 기록하면 AI가 부족한 영양소를 분석해 맞춤 식단을 추천합니다.

Result & Learning

  • "코딩알려주는누나" 강의를 마친 뒤 배운 것을 실전 서비스로 완성한 첫 팀 프로젝트입니다
  • PO로서 Figma 와이어프레임을 먼저 그리고 팀원과 방향을 맞추는 과정을 주도했습니다
  • TypeScript로 타입을 먼저 정의하는 습관이 형성되어 이후 프로젝트에도 기본으로 적용했습니다

Problem

  • 이력서를 여러 번 고쳐도 어디가 약한 건지 스스로 판단하기 어려웠습니다
  • ATS 관점에서 핵심 키워드가 부족한지 확인할 방법이 없었습니다
  • 어떤 문장을 어떻게 바꿔야 하는지 알려주는 도구도 없었습니다

Solution

  • PDF 업로드 후 정규표현식으로 섹션을 자동 분리하도록 구현했습니다
  • Puter.js AI가 섹션별 1~100점 채점과 문장 단위 수정 제안을 생성합니다
  • 이력서 키워드를 시각화하고, 업계 기준 대비 누락 키워드를 자동 제안합니다

  • AI 채점 — 각 섹션을 1~100점으로 평가하고 문장 단위 수정 제안을 생성합니다
  • 키워드 분석 — 이력서에 쓰인 키워드를 시각화하고 빠진 항목을 제안합니다
  • 상태 분리 — Zustand(UI 상태)와 React Query(AI 응답)로 역할을 분리했습니다
  • 응답 캐싱 — 같은 이력서를 다시 분석하면 캐시에서 즉시 결과를 돌려줍니다
  • 문구 개선 — "이 문장은 이렇게 바꾸세요" 수준의 구체적 수정안을 생성합니다

개발 중 겪은 문제

문제
  • AI 분석 결과·입력값·탭 상태가 여러 컴포넌트에 흩어져 있었음
  • props 깊이 3~4단계 → 데이터 변경 위치 추적 불가
  • 같은 이력서 재분석 시 AI 재호출 → 불필요한 API 비용 발생
해결
  • UI 상태(탭, 모달)는 Zustand, 서버 데이터는 React Query로 분리
  • React Query 캐싱으로 동일 이력서 재호출 제거
  • 상태 관리 위치 명확화 → 버그 탐색 시간 50% 감소
  • "클라이언트/서버 상태 분리"를 체감한 계기

사용자 경험 흐름

PDF 업로드→ 섹션별 점수 확인→ 수정 제안 확인→ 키워드 맵 확인→ 누락 키워드 제안

이력서를 올리면 AI가 즉시 섹션별 점수와 수정 방향을 알려주고, 키워드 맵에서 업계 기준 대비 빠진 키워드까지 확인할 수 있도록 구성했습니다.

Result & Learning

  • 클라이언트 상태와 서버 상태를 분리해서 관리하는 아키텍처를 직접 설계하고 적용했습니다
  • PDF 텍스트 추출부터 AI 채점, 키워드 분석까지 전체 파이프라인을 구축했습니다
  • 타입을 먼저 정의하고 개발하는 방식이 이후 프로젝트의 기본 원칙으로 자리잡았습니다

  • 동적 섹션 관리 — 항목을 자유롭게 추가·삭제하고 유니크 키로 안정 렌더링합니다

해결

  • 동적 항목에 uuid로 고유 키 부여 → 삭제 후에도 입력값 유지

경력, 학력 등 섹션을 선택하고 내용을 입력하면 오른쪽에서 즉시 이력서 형태로 확인할 수 있습니다. 필요에 따라 섹션을 자유롭게 추가·삭제할 수 있도록 설계했습니다.

Result & Learning

  • 여러 섹션의 폼 상태를 하나의 트리로 모아 관리하는 설계를 직접 해봤습니다

  • 상태 관리의 한계를 직접 느끼면서, 이후 Zustand 도입의 계기를 마련했습니다
