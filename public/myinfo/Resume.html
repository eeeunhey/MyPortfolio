성격 / 태도

안녕하세요 저는 경험 속에서 배우며 성장하는 김은혜입니다. 여러 직무와 프로젝트를 수행하면서 ‘완벽한 개인’보다 ‘함께 배우는 팀’이 더 큰 성과를 만든다는 것을 배웠습니다. 그래서 언제나 협력 속에서 배우고, 배운 것을 다시 나누려 노력합니다. 저는 누군가가 어려움을 겪고 있을 때 그냥 지나치지 못하는 성격입니다. 제가 도움이 될 수 있는 부분이 있다면 주저하지 않고 함께 해결하려 합니다. 이러한 성향은 여러 업무를 경험하며 더욱 확고해졌습니다. 좋은 동료들과 함께 일하면서 많은 도움을 받았고, 그에 보답하고 싶다는 마음으로 자연스럽게 협력에 참여했습니다. 협력으로 일을 빠르고 완성도 있게 마무리했던 경험은 팀워크의 가치를 깊이 깨닫게 해준 기억으로 남아 있습니다. 또한 과거에 제가 도왔던 분들이, 제가 어려움을 겪을 때 다시 손을 내밀어주셨던 경험을 통해 ‘협력의 선순환’ 을 배웠습니다. 저는 협력할 부분은 적극적으로 협력하되, 각자의 역할은 끝까지 책임감 있게 수행해야 한다고 생각합니다.

성장노력

성격

저는 문제를 마주했을 때 단순히 ‘왜 안 되는지’를 찾기보다 ‘서로 다른 기술과 아이디어를 어떻게 결합하면 더 나은 구조로 발전시킬 수 있을까’를 고민합니다.
새로운 기술을 배울 때도 단순히 기능을 익히는 데 그치지 않고, 그것이 기존 흐름과 어떤 방식으로 연결될 수 있는지를 직접 실험하며 이해하는 것을 좋아합니다.
이런 탐구의 과정 속에서 문제 상황은 오히려 새로운 조합을 시도할 기회가 되었고, 그 과정에서 개발의 진짜 즐거움을 느꼈습니다.

프로젝트를 진행하면서 단순히 기능이 동작하는 것보다, “왜 이 문제가 발생했고, 어떻게 구조적으로 개선할 수 있을까”를 중심으로 접근했습니다.
아래는 그 과정에서 실제로 느낀 문제와 이를 해결하기 위해 적용한 경험들입니다.

🟦 명확한 타입 정의로 유지보수 속도를 개선
  • React 프로젝트를 진행하며 컴포넌트 간 데이터 전달이 많아지자, 어떤 props가 어디서 넘어오고, 어떤 타입으로 쓰이는지를 추적하는 데 많은 시간을 쏟게 되었습니다. 데이터 구조가 조금만 바뀌어도 연쇄적으로 오류가 발생해 유지보수가 점점 어려워졌습니다.
  • 이런 반복적인 불편함을 줄이기 위해 TypeScript를 도입했습니다. 타입을 명확히 지정하니 IDE에서 자동완성·오류 탐지·함수 추적이 가능해졌고, 수정 시 영향 범위를 빠르게 파악할 수 있었습니다.
  • 특히 여러 API가 동시에 호출되는 화면에서 데이터 구조가 바뀌면 즉시 에러가 표시되어, 런타임 오류보다 훨씬 빠르게 문제를 인식하고 대응할 수 있었습니다.
    단순한 타입 검증을 넘어서, 코드의 흐름이 눈에 보이는 구조로 바뀌는 경험이었습니다.
🟩 Zustand 기반 전역 스토어 리팩터링으로 상태 불일치 개선
  • React 프로젝트가 커지면서 여러 컴포넌트가 각자 상태를 관리하게 되었고, 한쪽의 변경이 다른 화면에 반영되지 않거나 데이터가 어긋나는 문제가 발생했습니다.
  • 프로젝트가 커지면서 여러 컴포넌트가 동일한 데이터를 서로 다른 방식으로 관리하면서, 하나의 수정이 다른 화면에 반영되지 않거나 예기치 않은 재렌더링이 자주 일어났습니다.
  • 이를 해결하기 위해 Zustand를 도입해 상태를 한 곳에서 일관되게 관리하도록 구조를 변경했습니다. 전역 스토어로 감정·집중도·답변 데이터를 통합하고, 각 컴포넌트는 필요한 값만 구독하도록 분리하니 화면이 안정적으로 동기화되고 렌더 효율도 향상되었습니다.
  • 현재는 작은 기능부터 점진적으로 Zustand를 확장하며, 단일 소스·명확한 액션·불필요한 렌더 최소화 원칙을 중심으로 구조를 다듬고 있습니다.
🟧 DAO/DTO 패턴으로 DB 연동 구조 개선 및 유지보수성 향상
  • 콘솔 기반 ALBAIT 프로젝트에서는 게시글 등록과 회원 데이터를 DB와 직접 연동했습니다. 처음에는 SQL 쿼리를 코드 안에서 바로 호출했지만,
    기능이 늘어날수록 의존 관계가 복잡해지고 수정 시 오류가 잦았습니다.
  • 이를 해결하기 위해 DAO/DTO 패턴을 도입해 요청 → DB 처리 → 응답 과정을 단계별로 나눴습니다.
    이전에는 코드 곳곳에서 SQL을 직접 다뤘지만, 지금은 데이터 흐름이 한눈에 보이고 수정할 위치도 명확해졌습니다.
    데이터가 어디서 만들어지고, 어디로 전달되는지 흐름이 뚜렷해지면서 오류를 찾고 수정하는 시간이 크게 줄었습니다.

여러 프로젝트를 하면서, 데이터와 상태는 서로 영향을 주기 때문에 하나의 흐름 안에서 함께 관리해야 한다는 걸 배웠습니다.
저는 단순히 문제를 해결하는 데서 멈추지 않고, 구조를 분석하고 개선점을 찾아내 실제 성과로 이어지게 하는 과정에서 보람을 느낍니다.
새로운 기술이나 접근법이 등장할 때마다, 이를 기존 시스템에 효율적으로 결합해 속도나 유지보수성을 개선하는 방법을 꾸준히 실험해왔습니다.
작업할 때는 ‘이 코드가 전체 구조 속에서 어떤 가치를 만들어내는가’를 항상 고려하며, 문제를 부분적으로 보기보다, 전체 흐름 속에서 가장 합리적인 해결 방법을 찾으려 합니다.
이런 접근을 통해 기능을 안정적으로 개선하고, 복잡한 로직을 단순화해 팀의 효율을 높이기 위해 꾸준히 노력하고 있습니다.

앞으로는 구조적 사고를 기반으로, 성과를 단순히 내는 데 그치지 않고, 그 결과를 분석하고 보완하며 한 단계 더 나은 가치를 만들어가는 개발자로 성장하고자 합니다.
다양한 기술과 아이디어를 함께 엮어 더 큰 성과를 만들어가는 사람이 되고 싶습니다.
